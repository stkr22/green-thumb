# Development

Working on Green Thumb. For how the system fits together see
[architecture.md](architecture.md); for running it see [setup.md](setup.md).

## Repository structure

```
src/greenthumb/   FastAPI backend (config, db, models, schemas, auth, api/v1, services, main)
alembic/             database migrations
tests/               pytest suite (in-memory SQLite)
frontend/            React + TS SPA (src/, e2e/)
charts/green-thumb/   Helm chart (deployed via ArgoCD)
k8s/                 backend secret template (secret.example.yaml)
docs/                this documentation
Containerfile        backend image
docker-compose.yml   local full stack
```

## Getting started

Install and run both halves (full detail in [setup.md](setup.md)):

```bash
uv sync --group dev
uv run alembic upgrade head
uv run uvicorn greenthumb.main:app --reload     # backend on :8000

cd frontend && npm install && npm run dev           # SPA on :5173 (proxies to :8000)
```

For a login-free local loop, run the backend with `DEV_AUTH_BYPASS=true` and
visit `/auth/dev-login` (see
[administration.md](administration.md#dev-login-bypass-local-only)).

## Quality gate

Run after every change (matches CI expectations):

```bash
uv run ruff check --fix . && uv run ruff format . && uv run ty check
uv run pytest
cd frontend && npm run build      # tsc type-check + vite build
```

The backend test suite runs against in-memory SQLite with the `foreign_keys`
pragma enabled (so delete cascades behave correctly) and authenticates by
minting a real session cookie — see `tests/conftest.py`. New behaviour needs
tests; if outdated tests block a change, update them rather than working around.

## End-to-end tests (Playwright)

`frontend/e2e/` drives the real SPA against a real backend. Playwright starts
both servers itself — an isolated backend on a throwaway `greenthumb-e2e.db` with
`DEV_AUTH_BYPASS=true` (no Zitadel needed) plus the Vite dev server — and
authenticates once via `/auth/dev-login`, reused through `storageState`.

```bash
cd frontend
npx playwright install chromium    # first time only (on Linux CI also: npx playwright install-deps chromium)
npm run test:e2e
```

The runner needs `uv` on `PATH` (it launches the backend via `uv run`). Specs
create uniquely-named data so they're independent across runs.

## Regenerating the API client

The frontend's typed client (`frontend/src/api/types.gen.ts`) is generated from
the backend's OpenAPI schema. After changing routes or schemas:

```bash
SESSION_SECRET_KEY=x uv run python -c "import json; from greenthumb.main import app; open('openapi.json','w').write(json.dumps(app.openapi()))"
cd frontend && npm run generate-types
```

Hand-written hooks in `frontend/src/api/hooks/` wrap these types with TanStack
Query.

## Database migrations

```bash
uv run alembic upgrade head                       # apply
uv run alembic revision --autogenerate -m "..."   # author
```

SQLite can't `ALTER` columns/constraints in place, so `alembic/env.py` enables
**batch mode** (table rebuilds) for autogenerated migrations. Always verify a
new migration by running `alembic upgrade head` against a throwaway SQLite file.
The circular `plants.cover_photo_id` ↔ `plant_photos` FK is declared inline in
the initial migration (see [architecture.md](architecture.md#data-model)).

## Conventions

- **Async everywhere** — `async def` route handlers and service functions.
- **Thin handlers** — business logic lives in `services/`; one router module per
  domain in `api/v1/`.
- **Protected routes** use the `CurrentUser` / `get_current_user` dependency.
- **Datetimes** are UTC; normalise with `models.base.ensure_utc` when comparing
  values read back from SQLite.
- **Ruff `TC` rules are intentionally disabled** — FastAPI and SQLModel resolve
  annotations at runtime, so imports must not move into `TYPE_CHECKING` blocks.
- **Validate external input** with Pydantic / pydantic-settings.
- **Frontend:** functional components only, Tailwind utility classes, TanStack
  Query keys as `['resource', id?, 'sub-resource'?]`, errors surfaced via the
  toast context.
- **Document the _why_**, and update `docs/` after medium-to-large changes.
- **Commits:** conventional-commit style, explain the _why_, keep messages under
  300 characters.
